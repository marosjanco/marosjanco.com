Lexomat is an AI legal-research chat for Slovak and Czech law. Lawyers ask it questions; it answers with citations to specific paragraphs of specific acts. Those citations link to slov-lex.sk — the official Slovak legal portal — so a user can open the source and check every claim against the primary text. Early in the build I let the model generate those URLs as part of its answer. That was a mistake, and the way I fixed it generalises to almost any LLM system that has to cite external sources.

## The problem

Language models are very good at recognising the shape of a URL and surprisingly bad at constructing a valid one. A slov-lex URL encodes an act number, a year, an effective-date version, and sometimes a paragraph anchor. The model would produce something that looked exactly right — correct domain, correct path structure, a plausible date, a paragraph anchor in the correct format. Sometimes it was correct. Sometimes the effective date was invented, or the anchor pointed at a paragraph that did not exist in that version, or two digits of the act number were transposed.

The failure that mattered was not the obviously broken link. It was the link that looked authoritative, resolved to a real page, and showed the wrong provision. A lawyer following a citation to confirm a point does not expect the citation itself to be subtly wrong. In a legal product that is not a rough edge — it is the kind of defect that ends trust in the whole tool.

## Why "just prompt better" does not fix it

The instinct is to tighten the prompt. Give the model the URL schema. Add a few correct examples. Tell it, firmly, to be careful. I tried all of it. Each change moved the error rate around. None of it removed the error.

The reason is structural. Building a slov-lex URL from a citation is a *deterministic* transformation: given the act, the year, the version, and the paragraph, exactly one correct URL exists, derivable by a rule. Asking a probabilistic next-token model to perform a deterministic transformation is the wrong tool for the job, the same way asking it to add two seven-digit numbers is the wrong tool. It will be right most of the time. "Most of the time" is precisely the problem, because the failures are silent and the domain does not tolerate them.

There is a deeper point here. Prompting harder reduces the *rate* of an error class. Architecture can remove the error class. When the cost of a single failure is high, rate reduction is not enough — you have to make the failure unrepresentable.

## The architectural fix

Stop letting the model emit URLs at all. Have it emit *structured citation metadata*, and have the backend construct the URL from a verified index.

In practice the model's job became to produce a citation object:

```json
{
  "citation": {
    "act_number": 40,
    "year": 1964,
    "section": 588,
    "subsection": 1,
    "letter": "a"
  }
}
```

The backend takes that object and resolves it against a maintained index of slov-lex documents. It returns a verified, canonical URL — or, just as importantly, it reports the citation as unresolvable, and the answer degrades gracefully instead of presenting a confident broken link. The model decides *what* to cite. The system decides *how* to cite it, and whether the citation can be trusted at all.

Two parts of that did the real work, and they are worth separating.

### The index

The index is the source of truth. It maps `(act, year, version, paragraph)` to a canonical URL and records which paragraphs actually exist in which versions. It is built from the primary source, not from the model, and it is the thing that lets the backend say "no" instead of guessing. The model never sees the URL space; it only sees the citation it is responsible for, which is the part it is actually good at — reading legal text and identifying the provision being relied on.

### Pre-1993 disambiguation

Slovak and Czech law share a Czechoslovak federal past. Citations from before 1993 use formats that overlap with the later Slovak and Czech ones — the same act number can refer to different acts depending on which legal lineage you assume. This is exactly the kind of ambiguity you want resolved in code, with explicit rules and test cases, not left to a prompt to "be careful about". The resolution layer encodes the disambiguation rules and fails loudly when a reference is genuinely ambiguous, rather than silently picking one.

## What changed in production

Hallucinated URLs went away as a category — not "fewer", structurally absent, because the model can no longer produce that artifact. Other hallucination classes still exist; this technique does nothing for the model misstating the *content* of a provision. But an entire failure class disappeared by removing the model's ability to produce the thing that carried it.

The second-order effect was the more interesting one. Narrowing the model's responsibility made its remaining output easier to trust and easier to reason about. A model asked to do less, in a more structured way, fails in fewer places and fails more visibly.

## The general principle

Anything in your LLM output that has a deterministic correct form should be produced by code from structured model output, not by the model directly. URLs. Dates. Currency formatting. ID and record lookups. Arithmetic. SQL against a schema you control. The model's job is to decide *what*; the system's job is to construct *how*, and to refuse when it cannot.

This is not a Lexomat trick. It is a separation-of-concerns boundary that most production LLM systems get wrong in at least one place — usually because the demo worked and nobody went back to harden it. Finding those places is a fast audit: look for anywhere a deterministic artifact is being emitted inside free-form generation.

## Why this also makes evaluation tractable

There is a quieter benefit. When the citation is a structured field, you can evaluate it directly: parse it, check the act and paragraph against the index, and report accuracy as a number. You cannot do that when the URL is embedded in a free-form paragraph — there you are reduced to fuzzy string matching or another model grading prose. Pulling deterministic artifacts out of free-form generation does not just reduce hallucination; it gives you something you can actually test, which is the difference between "we think it's fine" and "we measure it every release".

## Closing

Most LLM systems I have looked at have at least one place where the model is being asked to emit a deterministic artifact inside free-form text. Finding those and refactoring them is some of the highest-leverage work in production LLM engineering. It does not require a better model. It requires drawing the boundary between what the model decides and what the system constructs in the right place — and then not letting the model cross it.
